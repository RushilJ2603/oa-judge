# OA Judge — Build Plan

A local, LeetCode/HackerRank-style judge for practising OA problems under either **visible-test
(LC)** or **hidden-test (OA)** conditions. Standalone: this folder is the root, with **no relation to
the DSA notes folder**. The notes stay frozen.

## What it is

You pick a problem, code a full solution in an in-browser editor, and the app **actually compiles and
runs it** against real test cases — not just displays code. Two modes:

- **LC mode** — all tests visible; on failure you see input, expected, and your output.
- **OA mode** — tests hidden (HirePro-style); you see only the verdict and which test index failed,
  plus a countdown timer and no post-submit reruns. Stray stdout fails you, exactly like a real OA.

Plus a **stress-test** button: random inputs are fed to both your code and a verified reference; the
first disagreement is **shrunk to the smallest failing input** and shown. This is the direct answer to
"I can't see the failing test."

## Decisions (locked)

| Question | Decision |
|---|---|
| Editor | In-app browser editor (plain, matches degraded OA editors) |
| LC vs OA difference | Purely whether tests are hidden; **same stdin/stdout harness** for both |
| Languages (v1) | **C++17** and **Python 3** |
| Existing LC/GfG questions | Show the pointer **and** import a local copy (LC shows you the failing test — the crutch we train off) |
| Source of truth | **The judge owns everything.** Statements live here; notes are not touched or linked |
| Features | Stress-testing vs reference · OA simulation mode · attempt history + revisit list · per-problem editorial bundled in-app |
| Build | Full v1, delegated to Gemini (agy) subagents with Claude orchestrating + verifying |

## Architecture

Python + Flask (both already installed; **no Node needed**). Frontend is vanilla HTML/CSS/JS served
by Flask — no build step. Code runs in subprocesses under CPU/memory/output rlimits.

```
oa-judge/
  problems/<id>/           # one folder per problem (see FORMAT.md)
     problem.json          # metadata, limits, languages, links, io mode
     statement.md          # canonical statement (judge owns this)
     editorial.md          # solution writeup + traps (bundled, replaces notes cross-link)
     stub.cpp / stub.py    # starter harness the user fills in
     reference.cpp/.py     # VERIFIED reference (Claude-owned, never delegated)
     generator.py          # random input generator for stress mode
     oracle.*  (optional)  # slow brute-force, when the reference itself is the fast one
     tests/
        sample/  NN.in NN.out    # visible in both modes
        hidden/  NN.in NN.out    # generated by running reference; shown only in LC mode
  app/
     server.py             # Flask app + JSON API (see API.md)
     runner/
        sandbox.py          # rlimits, timeouts, temp workspace
        run_cpp.py          # g++ compile + execute
        run_py.py           # python execute
        judge.py            # output comparison + verdict logic
        stress.py           # generator → reference vs user → shrink
        problems.py         # load/list problem packages
     static/ index.html app.js style.css
     data/ history.json     # attempt log
  add_problem.py           # scaffolder: creates a new problems/<id>/ skeleton
  start.sh / start.command / start.bat   # one-command launch → opens browser
  FORMAT.md  API.md  README.md  PLAN.md
  _migrated_raw/           # the original transcription material, for reference only
```

## Verdicts

`AC` accepted · `WA` wrong answer · `TLE` time limit · `MLE`/`RE` runtime (signal / bad_alloc /
nonzero exit) · `CE` compile error. Output comparison is whitespace-tokenized by default (trailing
newline/space tolerant); a problem may request exact-match.

## Build phases

- **Phase 0** *(Claude)* — skeleton, migrate questions, write PLAN/FORMAT/API. ✅
- **Phase 1** *(Claude)* — the engine (sandbox, runners, judge, stress, server) + **flipkart-q1
  packaged completely** as the template every other problem is cloned from. Boot and self-test.
- **Phase 2** *(Gemini, parallel, ≤20 agents)* —
  - Frontend UI built against `API.md`.
  - Remaining problems packaged (statement.md, editorial.md, problem.json, stub, generator) in
    batches against `FORMAT.md` + the flipkart-q1 template + the `_migrated_raw/` sources.
  - LC/GfG existence research → `links` per problem.
- **Phase 3** *(Claude)* — copy the **verified references** in (never delegated), generate hidden
  tests by running them, integrate agent output, and run the full gate: every reference must AC its
  own tests; deliberately wrong solutions must get the right non-AC verdict; stress mode must catch a
  planted bug.
- **Phase 4** *(Claude)* — `add_problem.py` scaffolder, one-command launch scripts, README, final
  smoke test.

## The friction contract (your side)

- **To practise:** run `start.sh` (or double-click `start.command`), the browser opens, pick a
  problem, code, Run / Submit. Toggle OA/LC per problem.
- **To add a problem you dictate:** you tell Claude the problem. If it's on LC/GfG, Claude gives the
  link and imports a local copy; if not, Claude scaffolds it with `add_problem.py`, writes the
  statement + a verified reference + tests. You never hand-edit JSON or wire anything.
